# Service Provider Patrol Full Coverage Design

## 背景

当前仓库已经具备服务商 Patrol 测试基础设施，并已初步落地以下能力：

- Patrol 测试底座、报告模型与结果输出
- 服务商首页与“我的”页面的基础可达性验收
- iOS `RunnerUITests` 集成与单设备串行运行
- 若干关键页面稳定锚点，如服务商首页、服务商“我的”、编辑套餐页

但现阶段覆盖范围仍偏“页面可达性”，尚未系统覆盖服务商角色下的真实交互行为，例如：

- 页面内按钮、筛选、列表项、返回、保存、删除
- 复杂表单输入与局部状态切换
- 上传入口、弹层、确认框、错误态、空态、重试
- 服务商主流程跳转后的依赖页稳定性

本设计旨在将服务商 Patrol 测试从“页面可达”升级为“分层全扫”的交互验收体系。

## 目标

- 为服务商主业务页面补齐 Patrol 自动化验收用例，覆盖稳定可测试的页面与交互点
- 将测试粒度细化到“交互点级别”，而不只停留在“页面是否进入”
- 对复杂交互采用分层验收，兼顾覆盖率、稳定性与维护成本
- 在 iOS 单指定模拟器下稳定运行，避免并行测试带来的 clone simulator 干扰
- 最终输出可读的服务商 Patrol 验收报告，明确 `PASS / FAIL / BLOCKED`

## 范围

### 核心页面

本次核心细测范围限定为服务商主业务的 5 组页面：

1. 服务商首页
2. 服务商“我的”
3. 套餐管理页

### 依赖页面

以下页面默认纳入“依赖页冒烟”范围，至少验证“可达 + 页面稳定 + 可返回”：

- 服务商我的信息页
- 资质认证流程页
- 财务结算页
- 订单详情页
- 聊天页
- 订单管理页
- 人才中心页

### 非目标范围

本次不扩展到以下内容：

- 非服务商角色页面
- 全站所有业务模块
- 纯视觉样式对比型测试
- 依赖系统相册/文件选择器内部 UI 的深度自动化
- 对 Patrol 之外的单元测试与 Widget 测试体系做全面重构

## 测试策略

### 总体策略

采用“分层全扫”策略：

- 对核心 5 页执行“可见交互全扫”
- 对依赖系统能力或强环境依赖的交互做“弱闭环验收”
- 对服务商主页面跳转出去的依赖页至少做“可达性 + 稳定锚点”验证

### 交互分层

#### A 类：强闭环交互

适用于可稳定验证操作前后状态变化的交互，要求 Patrol 断言明确的结果变化。

示例：

- 页面跳转与返回
- Tab 切换
- 条件筛选
- 列表项点击
- 保存草稿
- 发布
- 编辑
- 上架 / 下架
- 删除确认
- 表单输入后的值变化或状态变化

验收要求：

- 入口可见且可点击
- 点击或输入后到达目标状态
- 必要时支持返回上一层继续执行后续用例

#### B 类：弱闭环交互

适用于会触发系统弹层、权限弹窗、外部文件选择器或强依赖测试环境数据的交互。

示例：

- 封面上传
- 示例材料上传
- 相机 / 相册 / 文件选择入口
- 某些依赖后端动态数据的弹层选择

验收要求：

- 入口可见且可点击
- 系统前置弹层或页面内选择面板能正确弹出
- 若能在页面内观察到状态变化，则断言局部状态变化
- 若受系统级交互或环境限制，则记为 `BLOCKED`，并写入明确原因

#### C 类：依赖页冒烟

适用于由服务商主页面跳转出的页面。

示例：

- 聊天页
- 订单详情页
- 财务结算页
- 服务商我的信息页
- 资质认证页

验收要求：

- 可以从服务商页面进入
- 目标页能通过稳定锚点或回退文案确认到达
- 能稳定返回原页面继续后续测试

## 稳定性设计

### 稳定锚点策略

现有 `AppTestKeys` 仅覆盖部分页面根节点与少量操作按钮，后续需要补齐以下几类锚点：

- 页面根节点
- 页面级主入口按钮
- 返回按钮
- 筛选按钮
- 列表页主容器
- 关键表单区域
- 保存 / 发布 / 删除 / 上下架等主操作按钮
- 复杂子组件的局部区域

锚点命名原则：

- 统一放入 `AppTestKeys`
- 采用 `pageXxx`、`actionXxx`、`sectionXxx`、`fieldXxx` 形式
- 语义优先，不直接绑定视觉文案

### 路由与回退策略

- 优先使用稳定 key 判断目标页面已到达
- 若暂时没有稳定 key，则回退到页面核心文案
- 对自定义导航页避免优先依赖 `$.native.pressBack()`
- 对编辑套餐页等自定义返回场景，优先点击显式返回按钮

### 数据依赖策略

- 有稳定环境数据时，执行完整强闭环
- 无稳定环境数据时：
  - 允许验证空态 / 错误态 / 可点击性
  - 对缺少数据导致无法继续的用例记为 `BLOCKED`
- 所有 `BLOCKED` 必须写入明确原因，如：
  - 测试账号缺少套餐数据
  - 联系目标用户 ID 缺失
  - 上传依赖系统选择器，当前环境不支持自动确认

## 实施顺序

### 阶段 1：闭环当前 Task 3 基线

- 先确保服务商首页、服务商“我的”基础 Patrol 用例在单模拟器下稳定运行
- 清理并确认 iOS 串行执行配置有效

### 阶段 2：补齐核心页面锚点

- 为套餐管理、编辑套餐、签证页及依赖页补稳定测试 key
- 重点补页面根节点、主操作按钮、返回按钮、筛选入口

### 阶段 3：补齐套餐管理与签证页全扫用例

- 优先完成列表、筛选、跳转、按钮交互
- 同时覆盖空态、错误态、重试、刷新、加载更多

### 阶段 4：补齐编辑套餐页全扫用例

- 处理最复杂的表单、增删、选择、上传入口
- 对系统级上传能力按弱闭环策略实现

### 阶段 5：补齐首页与“我的”页深层交互

- 将现有“可达性”升级为“交互点级验收”
- 补齐资料卡、消息中心、订单区交互、AI 助手入口

### 阶段 6：全量重跑与报告收口

- 在指定 iOS 模拟器上串行执行服务商 Patrol 测试
- 汇总最终通过、失败、阻塞项
- 更新相关报告文档

## 风险与应对

### 风险 1：测试环境数据不稳定

应对：

- 用例允许 `BLOCKED`
- 对数据依赖明显的点预先校验前置条件
- 空态和错误态用例与有数据用例分开组织

### 风险 2：系统选择器与权限弹窗不稳定

应对：

- 上传类交互按弱闭环处理
- 只验证 Patrol 可控范围内的弹层、入口与页面状态变化

### 风险 3：复杂页面缺少稳定锚点

应对：

- 先补测试 key，再写 Patrol 用例
- 尽量避免直接依赖易变文案与层级位置

### 风险 4：单个测试文件过长导致脆弱

应对：

- 按页面和交互分组拆分用例
- 抽取公共等待、返回、结果记录 helper

## 成功标准

- 服务商核心 5 页的主要可见交互点均被 Patrol 覆盖
- 依赖页至少具备稳定的可达性与返回验证
- 上传、筛选、输入、按钮、删除、保存等关键交互均有自动化验收
- iOS 指定模拟器下不再拉起 clone 模拟器
- 全量测试运行后可输出可读的服务商 Patrol 报告
