# 求职者实名认证设计

## 背景

当前求职者端“我的”页面顶部资料卡只有昵称、手机号和已实名徽标，没有与设计稿一致的实名认证入口。现有工程里已经有：

- `GET /users/me` 返回 `isVerified`
- `POST /users/me/real-name-verify` 可提交实名认证

但没有求职者专用的实名认证页面。现有 `qualificationCertification` 是企业/服务商资质认证流，不适合直接复用到求职者端。

本次目标是在不影响企业/服务商认证流的前提下，为求职者端补齐：

- “我的”页实名认证提示与入口
- 求职者实名认证提交页
- 最小可用的表单校验、图片上传与提交成功回流

## 目标

- 在求职者“我的”页顶部资料卡中补齐实名认证相关展示和交互
- 新增求职者专用实名认证页，视觉和交互尽量贴近给定设计稿
- 直接复用现有 `UserService.realNameVerify()` 提交实名信息
- 仅使用 `isVerified` 作为状态来源，不扩展审核中、驳回等更多状态

## 非目标

- 不改造企业/服务商的资质认证流
- 不新增“实名认证详情查询”接口
- 不实现审核中、驳回态展示
- 不重构“我的信息”页面

## 现状结论

### 接口

- `GET /users/me`
  - 已返回 `isVerified`
- `POST /users/me/real-name-verify`
  - 已存在
  - 请求体字段为：
    - `realName`
    - `idCardNumber`
    - `idCardFrontUrl`
    - `idCardBackUrl`

### 前端

- `CurrentUserViewData` 已映射 `isVerified`
- 求职者“我的”页已有昵称右侧 `已实名` 小徽标
- 没有求职者专用实名认证页
- 现有 `qualificationCertification` 仅支持：
  - `serviceProvider`
  - `company`

## 用户交互设计

### 我的页资料卡

在求职者“我的”页顶部资料卡中保留现有头像、昵称、手机号和箭头，新增实名认证提示区：

- 昵称右侧继续显示 `已实名` 徽标
- 昵称下方、手机号附近新增一行实名认证提示
- 当 `isVerified == true`
  - 展示文案：`已完成实名认证`
- 当 `isVerified == false`
  - 展示文案：`您还未实名，点击去实名认证`

交互规则：

- 点击实名认证提示区，进入求职者实名认证页
- 不改变“我的信息”页面本身的数据结构
- 不把企业/服务商资质认证页作为求职者入口

### 求职者实名认证页

页面采用单页表单，贴近设计稿，包含：

- 标题：`实名认证`
- 姓名输入框
- 身份证号输入框
- 身份证验证上传区
  - 上传国徽面
  - 上传人像面
- 底部主按钮：`同意并提交`
- 底部提示文案：说明身份证照片用途

### Task 2 视觉对齐约束

Task 2 以本地导出的 Figma 设计为最高优先级，按以下资源收敛页面视觉：

- 样式参考：`/Users/linwei/BlueHub/.figma/129_16461/index.module.scss`

本轮实现要求：

- 页面整体背景、顶部标题区、白色表单卡片、底部说明文案和固定底部按钮的层级尽量按截图 1:1 还原
- 姓名、身份证号字段采用更接近设计稿的“行式输入”视觉，而不是大面积灰底输入框
- 身份证验证区使用设计稿中的上下结构：标题与副文案在上，左右两张上传示意图居中排列，底部再展示 `上传国徽面`、`上传人像面`
- 空态示意图直接使用现有资源：
  - `assets/images/qualification_id_emblem.png`
  - `assets/images/qualification_id_portrait.png`
- 不再使用通用加号图标或通用上传占位卡风格
- 继续保留 Task 2 已完成的本地校验与图片选择逻辑，只调整视觉呈现和布局细节

交互流程：

1. 用户从“我的”页点击实名认证入口
2. 进入实名认证页
3. 录入姓名、身份证号
4. 选择或拍摄身份证正反面
5. 点击提交
6. 本地校验通过后，调用接口提交
7. 成功后返回上一页，并刷新登录用户信息
8. 资料卡根据新的 `isVerified` 更新显示

## 路由设计

新增独立路由：

- `RoutePaths.jobSeekerRealNameVerification`

并在 `app_router.dart` 中注册对应页面。

入口仅从求职者“我的”页跳转到这个新路由。

## 数据与状态设计

### 表单字段

- `realName`
- `idCardNumber`
- `idCardFront`
- `idCardBack`

### 页面状态

- `isSubmitting`
- `frontImage`
- `backImage`
- 字段级错误文案状态（用于页面内可见错误提示）

### 校验规则

- 姓名不能为空
- 身份证号不能为空
- 国徽面必须上传
- 人像面必须上传

当前版本只做最小校验，不额外加入复杂身份证格式校验。

## 提交流程设计

### Task 2 当前范围

- 点击 `同意并提交` 时执行本地最小校验
- 缺项时在页面内直接展示错误文案，并阻止继续提交
- 点击上传区时复用 `UploadPickerUtils.pickImagesWithSourceSheet()` 打开来源选择
- 选图成功后仅更新页面本地展示状态，不执行真实上传

### 后续任务范围

后续任务再补：

- 图片真实上传
- `UserService.realNameVerify()` 提交
- 登录态刷新和返回页回流

### 提交

调用：

- `UserService.realNameVerify(request: RealNameVerifyBO(...))`

请求体：

- `realName`
- `idCardNumber`
- `idCardFrontUrl`
- `idCardBackUrl`

### 成功处理

- 展示成功提示
- 返回上一页
- 触发当前登录用户信息刷新
- 资料卡重新读取 `isVerified`

### 失败处理

- 保持停留在当前页
- 展示接口错误或通用失败提示
- 不清空用户已填写内容

## 错误处理

- 图片选择取消：保持静默，不提示错误
- 图片上传失败：提示失败并允许重试
- 提交失败：显示服务端错误信息，若无则显示通用失败提示
- 页面返回后不做额外复杂状态恢复，本次仅保证当前会话内交互正确

## 测试策略

高价值测试优先覆盖：

- `我的` 页实名认证入口展示逻辑
  - 已实名时展示已实名状态
  - 未实名时展示引导文案
- 点击入口会跳转到新的求职者实名认证页
- 表单缺项时会阻止提交并提示
- Task 2 的视觉骨架应稳定渲染出标题、行式输入、身份证验证区和固定底部提交按钮
- 图片选择行为保留现有工具接入点，后续再补上传和提交流测试

## 风险与取舍

- 因后端没有实名认证详情查询接口，本次无法做“已提交后回显实名信息”能力
- 因当前只依赖 `isVerified`，无法区分审核中与驳回态
- 为避免误用企业/服务商认证流，本次新增求职者专用页面，而不是硬复用已有三步认证页

## 验收标准

- 求职者“我的”页资料卡出现实名认证提示区
- 点击提示区进入求职者专用实名认证页
- 页面包含姓名、身份证号、正反面上传与提交按钮
- 页面视觉结构与提供的 Figma 截图尽量一致
- 上传空态使用身份证国徽面与人像面示意图资源
- 点击提交缺项时会在页面内出现可见错误文案并阻止继续提交
