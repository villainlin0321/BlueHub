# 编辑页键盘收起统一设计

## 背景

当前项目中，部分编辑页已经具备“点击空白区域收起键盘”的能力，但实现方式不统一：

- 一部分页面使用 `TapBlankToDismissKeyboard`
- 一部分页面只在提交前手动 `unfocus()`
- 一部分长表单具备滚动收键盘能力，但没有形成统一规范
- 个别页面存在自定义点击空白失焦逻辑，与共享组件并存

这会导致实际交互体验不一致，用户在编辑页输入后点击别处时，键盘可能没有稳定收起，甚至在切换别的可点击区域后再次弹起。

本次设计目标是以最低风险的方式，统一编辑页的焦点收起行为，让“点击空白区域”和“拖动长表单”都能稳定收起当前输入焦点。

## 目标

- 统一编辑页的键盘收起交互，减少页面间体验差异
- 让用户在编辑页点击非输入区域时稳定失焦
- 让长表单在拖动滚动时同步收起键盘
- 尽量复用现有共享组件和页面结构，降低改造风险
- 为该行为补充自动验证，覆盖 `Widget` 与 `Patrol` 两层

## 非目标

- 不扩展到搜索页、聊天页和纯输入弹层
- 不在本轮全量改造所有 `TextField/TextFormField` 的 `onTapOutside`
- 不为了统一焦点处理而重构页面业务结构
- 不把本轮目标扩大为“所有输入场景通用焦点框架”

## 范围

### 纳入范围

本轮仅处理明确属于资料编辑、表单编辑、信息编辑的页面，包括但不限于：

- `lib/features/me/presentation/add_work_experience_page.dart`
- `lib/features/me/presentation/add_education_experience_page.dart`
- `lib/features/me/presentation/my_resume_editor_page.dart`
- `lib/features/me/presentation/self_evaluation_page.dart`
- `lib/features/auth/presentation/qualification_certification_page.dart`
- `lib/features/auth/presentation/qualification_certification_step_three_page.dart`
- `lib/features/visa/presentation/widgets/edit_visa_package_page_view.dart`

### 不纳入范围

- 搜索页，如岗位搜索、服务商搜索
- 聊天输入页
- 底部弹层、弹窗类临时表单
- 明显依赖复杂 `FocusNode` 编排的特殊交互页，除非它同时也是本轮编辑页目标且可以低风险接入

## 现状结论

### 已有共享能力

项目已经存在统一的页面级点击空白收键盘组件：

- `lib/shared/widgets/tap_blank_to_dismiss_keyboard.dart`

其当前行为是：

- 通过 `GestureDetector`
- 在点击空白区域时读取 `FocusScope`
- 当当前节点不是主焦点时执行 `unfocus()`

这是本轮统一方案的基础。

### 已有页面模式

当前项目里已经出现三种相关模式：

1. 页面级空白点击失焦
2. 滚动容器拖动收键盘
3. 字段级 `onTapOutside` 失焦

其中，页面级与滚动级已经足以覆盖本轮需求；字段级 `onTapOutside` 作为更强兜底能力，暂不作为本轮统一范围。

## 设计方案

### 总体方案

采用“页面级 + 滚动级”的统一策略：

1. 页面根部统一接入 `TapBlankToDismissKeyboard`
2. 长表单滚动容器统一接入 `keyboardDismissBehavior: ScrollViewKeyboardDismissBehavior.onDrag`

这样可以覆盖两类最常见的编辑行为：

- 用户点击输入框外部区域时收起键盘
- 用户开始拖动表单时收起键盘

### 页面级策略

所有编辑页在最外层布局统一包裹 `TapBlankToDismissKeyboard`，用于收敛“点击空白失焦”行为。

约束如下：

- 如果页面已经使用该共享组件，则保持不变
- 如果页面有等价的自定义点击空白实现，优先收敛成共享组件或保持等价逻辑，不叠加两层相互竞争的手势
- 如果页面最外层已经存在对点击手势敏感的区域，需要优先确认是否会影响原有点击行为，再决定包裹位置

### 滚动级策略

所有长表单类编辑页，只要内部使用以下滚动容器之一，就统一补齐拖动收键盘配置：

- `SingleChildScrollView`
- `ListView`
- `CustomScrollView`

统一行为：

- `keyboardDismissBehavior: ScrollViewKeyboardDismissBehavior.onDrag`

约束如下：

- 仅对真实承载表单内容的滚动容器补齐，不在无关列表页扩大使用
- 页面中若已有该配置则保持不变
- 若页面存在多层嵌套滚动，只给实际响应用户拖动的主滚动容器加配置

### 特殊页面策略

对于已有特殊焦点控制的编辑页，采用“最小侵入”原则：

- 如果页面需要显式管理 `FocusNode`，本轮只补页面级空白点击和主滚动拖动收起，不重写其字段焦点编排
- 如果页面已有自定义点击空白逻辑但行为正确，可不强制替换，只要最终交互符合统一预期即可
- 如果某个页面经验证仍存在局部异常，再针对该页单独补 `onTapOutside`，不提前扩大改造面

## 代码落地方式

### 共享组件

本轮优先复用现有共享组件：

- `lib/shared/widgets/tap_blank_to_dismiss_keyboard.dart`

若需要增强，仅允许做低风险调整，例如：

- 补充更明确的注释
- 为后续测试增加稳定的行为说明

本轮不新增新的“全局表单容器基类”。

### 页面改造方式

逐页检查编辑页结构，按以下顺序接入：

1. 判断页面是否已具备页面级点击空白失焦
2. 判断主表单滚动容器是否已具备 `onDrag` 收键盘
3. 对缺失项进行最小补齐
4. 避免与原有自定义手势冲突

### 代码改动边界

允许修改：

- 编辑页最外层包裹结构
- 编辑页主滚动容器参数
- 少量与焦点收起直接相关的共享组件

不允许借机修改：

- 页面业务提交流程
- 字段校验逻辑
- 路由结构
- 非编辑页交互

## 自动验证设计

本轮自动验证采用两层策略：

- `Widget` 测试负责验证焦点是否失去
- `Patrol` 测试负责验证真实页面交互链路上的键盘收起体验

### Widget 测试

#### 目标

验证以下逻辑在 Flutter 测试环境中稳定成立：

- 点击空白区域后，当前输入框失焦
- 拖动表单滚动容器后，当前输入框失焦

#### 测试落点

建议新增两类 `Widget` 测试：

1. 共享组件级测试
2. 代表性编辑页测试

建议文件：

- `test/shared/widgets/tap_blank_to_dismiss_keyboard_test.dart`
- 代表性页面对应的测试文件，例如已有同模块测试文件中追加焦点相关用例，或新增 `test/features/me/..._test.dart`

#### 断言方式

在 `Widget` 测试中，不直接断言系统键盘 UI，而是断言焦点状态变化，例如：

- `FocusNode.hasFocus == false`
- `primaryFocus == null`
- 当前 `EditableText` 不再持有主焦点

#### 代表页面选择

优先选择一个结构稳定、可重复输入的编辑页作为代表页，建议从以下页面中二选一：

- `my_resume_editor_page.dart`
- `self_evaluation_page.dart`

选择标准：

- 有明确输入框
- 有明确空白区域
- 有主滚动容器
- 不强依赖复杂异步前置条件

### Patrol 测试

#### 目标

在真实设备/模拟器环境中验证：

- 进入编辑页后可以稳定唤起输入焦点
- 点击页面其它区域后键盘收起
- 拖动页面后键盘收起
- 收起后不会因为点击其它非输入交互区域而再次异常弹起

#### 测试目录

项目当前已具备 `Patrol` 基础设施，测试目录位于：

- `patrol_test/`

因此本轮不新建 `integration_test` 体系，而是沿用现有 `Patrol` 目录和辅助方法。

#### 用例建议

建议新增 1 条代表性 Patrol 用例，覆盖 1 个真实编辑页的完整链路，避免一开始大面积铺开导致稳定性下降。

建议优先页面：

- `my_resume_editor_page.dart`
或
- `edit_visa_package_page_view.dart`

若选择后者，需要确认登录角色、路由入口和页面锚点已经具备稳定测试前置条件。

#### Patrol 验收口径

Patrol 用例不要求验证系统键盘每一帧动画，而要求验证真实交互结果，例如：

- 输入后页面处于可编辑状态
- 点击空白后焦点离开输入区
- 拖动后页面仍可继续操作，且不出现键盘残留导致遮挡
- 后续点击其它按钮或区域时，不出现“键盘又被异常拉起”的问题

如遇系统级键盘状态在模拟器中不可稳定直接读取，可采用页面焦点状态、输入光标消失、后续交互可执行等可观察结果作为替代断言。

## 验证策略

### 手工验证

除自动化外，仍需要对改造页执行最小手工回归：

1. 进入编辑页
2. 点击输入框，唤起键盘
3. 点击页面空白区域，确认键盘收起
4. 再次点击其它非输入控件，确认键盘不会异常重新弹出
5. 重新聚焦输入框后，拖动页面，确认键盘收起

### 自动验证执行策略

实施完成后至少执行：

- 相关 `Widget` 测试
- 新增 Patrol 用例
- 涉及页面已有测试文件的回归执行

不要求在本轮把所有编辑页都补成 Patrol 用例，只要求先建立一条稳定的真实交互验收链路。

## 风险与控制

### 风险一：手势冲突

风险：

- 页面级 `GestureDetector` 可能与原有点击区域冲突

控制方式：

- 优先复用现有 `TapBlankToDismissKeyboard`
- 仅在合适层级包裹，避免覆盖已有明确点击逻辑
- 对已有自定义实现页先做局部收敛，不做粗暴双重包裹

### 风险二：滚动行为变化

风险：

- 补 `keyboardDismissBehavior` 后，某些页的滚动体验可能和此前略有不同

控制方式：

- 仅对编辑页主滚动容器启用
- 不在列表展示页或非表单页扩散使用
- 通过代表页自动化和手工回归双重确认

### 风险三：Patrol 稳定性不足

风险：

- 真机/模拟器下系统键盘状态不一定总能直接读取

控制方式：

- Patrol 以“用户可观察结果”为主要验收口径
- 首轮只落 1 条代表用例，先验证稳定性
- 如系统级断言不稳定，则退回焦点状态和后续交互结果断言

## 验收标准

- 本轮范围内编辑页都具备页面级点击空白收键盘能力，或已有等价实现
- 本轮范围内长表单编辑页都具备拖动滚动收键盘能力
- 点击非输入区域后，页面输入焦点可稳定释放
- 键盘收起后，不再出现明显的“点击别的地方又弹起来”的异常体验
- 至少新增 1 个共享或代表页 `Widget` 测试，验证失焦行为
- 至少新增 1 条代表性 `Patrol` 用例，验证真实编辑页交互链路
