# 资质认证最终统一提交设计

## 背景

当前资质认证流程采用“即选即传”策略：

- 第一步身份证选图后，立即调用资质上传接口。
- 第二步营业执照/特许经验许可选图后，立即调用资质上传接口。
- 第三步点击最终提交时，仅调用基础资料更新接口。

这与目标交互不一致。新的目标是：

- 选图阶段只在本地暂存图片与表单状态，不调用资质上传接口。
- 最终提交时，将 4 张图片中实际存在的图片一次性组装为 `docs` 数组，统一调用资质上传接口。
- 资质上传成功后，才调用基础资料更新接口。
- 必填图片未选中或未完成成功上传时，不允许进入下一步。
- 侧滑返回禁用；保留左上角返回按钮；当存在未保存修改时，返回前弹确认框。

## 目标

重构服务商/企业资质认证流程的上传时机与步骤校验逻辑，使其满足以下要求：

1. 第一步：
   - 两张身份证均为必填。
   - 未满足条件时，不允许进入第二步。
2. 第二步：
   - 营业执照必填。
   - 特许经验许可选填。
   - 未满足条件时，不允许进入第三步。
3. 第三步最终提交：
   - 不再复用“即选即传”。
   - 先批量上传并登记 `docs`。
   - `docs` 提交成功后，再提交基础资料。
4. 返回交互：
   - 禁用 iOS 侧滑返回。
   - 左上角返回仍保留。
   - 表单或图片有改动时，返回前弹确认框。

## 非目标

- 不改动求职者实名认证流程。
- 不改动认证流程之外的上传组件。
- 不扩散到其他业务模块的返回拦截逻辑。
- 不改变现有资质预览的视觉样式，仅调整上传与校验时序。

## 接口契约

### 资质上传接口

- 接口：`POST /visa-providers/me/qualifications`
- 文档位置：[0510api.json](file:///Users/linwei/BlueHub/docs/api/0510api.json#L2552-L2633)
- 请求体：`UploadQualificationDocsBO`
- 提交方式：最终提交时一次性传入 `docs` 数组

### 服务商资料更新接口

- 接口：`PUT /visa-providers/me`
- 文档位置：[0510api.json](file:///Users/linwei/BlueHub/docs/api/0510api.json#L193-L272)
- 请求体：`UpdateVisaProviderBO`
- 约束：该接口不承载图片文档字段，只承载基础资料与服务信息

企业端沿用同样思路：

- 先调用企业端资质上传接口
- 成功后再调用企业资料更新接口

## 现状问题

### 1. 上传时机错误

当前 `QualificationUploadHelper.uploadQualificationImage()` 在选图后立即：

1. 上传文件到文件服务
2. 调用资质上传接口登记文档

这导致图片在步骤一/二就已经提交到后端，和“最终统一提交”目标冲突。

### 2. 下一步校验缺失

当前第一页和第二页缺少针对必填图片的严格校验：

- 身份证正反面未完整准备时，仍可能进入下一步
- 营业执照未准备完成时，仍可能进入第三步
- 页面未对“本地已选中但未真正上传成功”的状态做流程级拦截

### 3. 返回交互不符合要求

当前流程页未统一禁用侧滑返回，且返回前缺少“有修改时确认退出”的保护。

## 设计方案

### 一、上传策略改为“本地暂存 + 最终统一提交”

步骤一与步骤二选图时：

- 仅保存本地 `PickedUploadFile`
- 将本地图片信息写入 `draft`
- 不调用资质上传接口
- 不生成远端 `fileId/fileUrl`

步骤三点击“最终提交”时：

1. 收集 `draft` 中所有本地图片
2. 逐张上传到文件服务，获取 `fileId/fileUrl`
3. 组装 `docs` 数组
4. 调用资质上传接口提交 `docs`
5. 资质上传成功后，再调用基础资料更新接口

这样可以保证：

- 图片与表单信息的最终提交时机一致
- 提交顺序满足“先 docs，后 profile”
- 若图片上传失败，基础资料不会被错误提交

### 二、必填图片下一步校验

#### 第一步校验规则

第一页点击“下一步”时必须满足：

- 国徽面图片已选择
- 人像面图片已选择

如果任一缺失：

- 阻止进入第二步
- 提示用户对应必填图片未完成

#### 第二步校验规则

第二步点击“下一步”时必须满足：

- 营业执照已选择

特许经验许可允许为空。

若营业执照缺失：

- 阻止进入第三步
- 给出明确提示

### 三、最终提交状态机

最终提交阶段采用严格串行流程：

1. 校验第三步表单
2. 校验必填图片是否已经在本地准备完毕
3. 上传图片文件到文件服务
4. 批量提交 `docs`
5. 提交基础资料
6. 成功后进入结果页

任一步失败都立即中断：

- 当前页保留
- 不继续后续接口
- 显示失败提示

采用串行而不是并发上传，理由如下：

- 错误定位更直接
- 更容易和现有错误提示体系对齐
- 更符合“先确保图片上传成功，再提交认证信息”的业务表达

### 四、返回与离开保护

第一页、第二页、第三页统一使用同一套离开保护思路：

- 禁用侧滑返回
- 保留左上角返回按钮
- 使用页面初始快照与当前状态比对，判断是否有未保存修改
- 若存在修改，则点击返回时先弹确认框
- 用户确认后才真正离开页面

“有修改”的判定包含：

- 文本字段变化
- 国家/地区选择变化
- 必填或选填图片变化

## 数据模型调整

`QualificationCertificationDraft` 继续作为流程内单一状态源，但语义需要明确区分两种图片状态：

1. 历史已提交图片
   - 有远端 `fileId/fileUrl`
   - 用于回显
2. 本次新选择但尚未提交的图片
   - 仅有本地路径
   - 用于预览与最终统一上传

最终提交时，如果用户未重新选择图片，但草稿里已存在历史远端图片，则该图片应直接参与 `docs` 组装，而不重复上传文件。

如果用户重新选择了某张图片，则优先使用新的本地文件执行上传并覆盖旧的远端文档信息。

## 测试策略

### 单元/组件测试

新增或更新 `flutter_test` 用例，覆盖：

1. 第一步：
   - 身份证任一缺失时，不能进入下一步
   - 身份证齐全时，可以进入下一步
2. 第二步：
   - 营业执照缺失时，不能进入下一步
   - 特许经验许可缺失时，仍可进入下一步
3. 第三步：
   - 最终提交顺序为：文件上传 -> docs 提交 -> profile 提交
   - `docs` 提交失败时，不会调用 profile 提交
   - profile 提交失败时，会正确提示失败
4. 返回交互：
   - 有修改时返回弹确认框
   - 无修改时允许直接返回

### 人工验证

至少覆盖以下场景：

1. 只填文字，不选身份证，第一页不能下一步
2. 身份证只选一张，第一页不能下一步
3. 第二步不选营业执照，不能下一步
4. 第二步不选特许经验许可，可以下一步
5. 最终提交时成功先提交 docs，再提交资料
6. 上传失败时停留当前页且资料不提交
7. 侧滑返回失效，左上角返回正常
8. 有改动时返回弹确认框

## 风险与处理

### 风险 1：历史已提交图片与新本地图片并存

处理：

- 以“最新用户选择”为准
- 若用户重新选择图片，则新图片覆盖旧图片参与最终提交

### 风险 2：文件上传成功但 docs 提交失败

处理：

- 视为最终提交失败
- 不再继续提交资料
- 页面停留，允许用户重试

### 风险 3：docs 提交成功但 profile 提交失败

处理：

- 视为整体失败
- 页面停留并提示
- 允许用户再次提交资料

该场景会造成 docs 已成功但 profile 未成功，这是后端双接口模式天然存在的中间态；本次不引入后端回滚能力。

## 验收标准

- 选图阶段不再调用资质上传接口
- 最终提交阶段批量提交 `docs`
- `docs` 成功后才调用资料更新接口
- 第一步身份证双图必填校验生效
- 第二步营业执照必填校验生效
- 特许经验许可保持选填
- 侧滑返回禁用
- 左上角返回保留
- 有改动返回时弹确认框
- 自动化测试通过，诊断无新增错误
