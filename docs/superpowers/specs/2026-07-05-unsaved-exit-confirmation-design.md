# 未保存返回确认弹窗设计

## 背景

当前项目中的多个编辑页、步骤表单页存在直接返回的行为。用户在页面内修改内容后，如果误触左上角返回、系统返回手势或系统返回键，当前输入内容会直接丢失，缺少统一的确认提醒。

本次需求要求：

- 编辑页面返回时，如果页面内容发生实际改动，则弹出统一确认弹窗。
- 弹窗使用现有 `AppDialog` 承载，视觉样式参考 Figma `Group 16`。
- 触发条件为“实际改动后才弹”，而不是“页面本来有内容就弹”。
- 需要同时覆盖页面返回按钮与系统级返回链路。

## 目标

- 统一未保存返回确认的交互与视觉样式。
- 为编辑页提供可复用的返回拦截能力，避免业务页面重复实现弹窗。
- 在不引入大规模重构的前提下，兼容新建态和编辑态页面。
- 为后续其他编辑页接入同一能力预留明确扩展模式。

## 非目标

- 不在本次需求中引入自动保存。
- 不统一重构所有编辑页为单一表单基类。
- 不修改现有提交、保存草稿、发布等业务流程。

## 方案概览

采用“公共未保存确认 helper + 页面自定义脏数据快照”的方案。

实现分为两层：

1. 公共层提供统一的未保存确认弹窗与离页确认方法。
2. 业务页各自提供当前页面的 `hasUnsavedChanges` 判断，并在返回入口处接入统一确认逻辑。

该方案可以在保持页面字段差异与现有结构的前提下，实现统一的交互表现和可维护的扩展方式。

## 弹窗设计

### 视觉与文案

- 主文案：`现在退出，内容将不会保存`
- 次按钮：`取消`
- 主按钮：`确定`
- 容器宽度、圆角、按钮布局、按钮颜色沿用现有 `AppDialog` 视觉体系。
- 文案层级按 Figma 处理为单行主提示，不额外展示灰色说明文案。

### 交互

- 用户点击 `取消`：关闭弹窗，停留在当前页面。
- 用户点击 `确定`：关闭弹窗，并执行真正的返回。
- 当页面没有未保存变更时，不弹窗，直接返回。

## 公共层设计

### 新增能力

公共层新增一个未保存确认 helper，职责如下：

- 统一弹出“未保存返回确认”弹窗。
- 统一返回 `bool` 表示当前是否允许离开页面。
- 让业务页只关注“是否有改动”，不需要重复实现弹窗样式。

### 建议接口

```dart
Future<bool> showUnsavedChangesExitDialog(BuildContext context);

Future<bool> confirmDiscardChangesIfNeeded({
  required BuildContext context,
  required bool hasUnsavedChanges,
});
```

说明：

- `showUnsavedChangesExitDialog` 负责纯弹窗展示。
- `confirmDiscardChangesIfNeeded` 负责根据 `hasUnsavedChanges` 决定是否直接放行。

## 页面接入设计

### 返回链路

每个接入页面同时拦截以下两条链路：

- 页面显式返回按钮，如 `AppBar.leading` 或自定义 header 的返回按钮。
- 系统返回链路，如 iOS 返回手势、Android 返回键，通过 `PopScope` 接入。

### 脏数据判断原则

统一采用“初始快照 vs 当前快照”模式。

- 新建态页面：初始快照为页面默认空白值。
- 编辑态页面：在接口回填完成后记录一次初始快照。
- 当前快照：在用户尝试返回时即时采集当前 UI 状态。
- 只有两者不一致时，才认为存在未保存改动。

### 快照内容

页面快照至少覆盖会影响提交结果的字段，不包含纯展示态数据。

以签证套餐编辑页为例，快照需覆盖：

- 套餐总名称
- 服务国家
- 签证类型
- 预计工作日
- 币种
- 封面图
- 每个档位的名称、价格、描述、服务标签、自定义服务
- 材料显隐、材料必填状态、材料类型、材料描述、示例文件

以资质认证步骤页为例，快照需覆盖：

- 第二步：营业执照、特许经验许可的已上传状态与文件标识
- 第三步：服务国家、从业年限

## 首批接入页面

本次先接入需求明确且当前用户正在使用的页面：

- `EditVisaPackagePage`
- `QualificationCertificationStepTwoPage`
- `QualificationCertificationStepThreePage`

后续其他编辑页可以沿用同样模式扩展：

- 页面内实现本页快照结构与比对方法。
- 返回入口统一调用公共 helper。

## 兼容性与边界

### 编辑态异步回填

编辑态页面存在接口异步加载详情的情况，例如签证套餐编辑页。

处理规则：

- 详情加载中不记录初始快照。
- 详情加载成功并完成表单回填后，再记录初始快照。
- 若加载失败，则不进入“有改动”判断流程。

### 上传中状态

页面中若存在图片或文件上传，需要将上传结果纳入快照，但上传进度本身不作为快照字段。

处理规则：

- 成功态记录文件标识或稳定 URL。
- 失败态或待上传态按当前页面状态参与判断，这意味着用户新增但未成功上传的文件也会被视为改动。

### 提交成功后返回

保存、发布、提交审核成功后的正常离页，不应再次弹出未保存确认。

处理规则：

- 提交成功后设置一次性放行标记，或在成功回调前先重置基线快照，再执行返回。

## 测试与验收

### 静态验证

- 运行 Flutter 分析或 IDE 诊断，确保新增 helper 与页面接入不引入编译问题。

### 手动验收

使用项目默认模拟器进行实际运行验收：

- iOS 模拟器 UDID：`8E3DF829-E72C-4B7D-8157-AAC88A8926B8`

验收点：

1. 页面未修改时，点击返回直接离开。
2. 页面修改后，点击左上角返回弹出确认框。
3. 页面修改后，触发系统返回链路同样弹出确认框。
4. 点击 `取消` 后留在当前页面。
5. 点击 `确定` 后离开当前页面。
6. 编辑态页面刚加载完成、未改动时不误弹。
7. 提交成功后的业务返回不被误拦截。

## 风险

- 不同页面的字段结构差异较大，快照字段遗漏会导致误判。
- 如果页面内部存在临时 UI 状态未与提交数据严格对应，可能导致“看起来改了但未计入快照”或“看起来没改但被判定改动”。

## 推荐实施顺序

1. 增强或复用 `AppDialog`，落地统一未保存确认弹窗 helper。
2. 先接入 `QualificationCertificationStepThreePage`，验证基础文本与选择项场景。
3. 接入 `QualificationCertificationStepTwoPage`，验证上传文件场景。
4. 接入 `EditVisaPackagePage`，验证复杂嵌套表单场景。
5. 在默认模拟器上完成实际交互验收。
