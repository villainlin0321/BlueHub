# BlueHub 日志系统设计

## 1. 背景与目标

当前项目已经具备基础日志能力：

- `AppLogger` 负责控制台输出、本地文件落盘、日志脱敏与裁剪。
- `AppLogInterceptor` 已覆盖基础 HTTP 请求、响应、异常日志。
- 少量核心流程已经手工补充业务日志，如应用启动、登录态恢复、路由重定向、岗位发布等。

现阶段的主要问题：

- 日志覆盖不均匀，不同模块记录粒度差异较大。
- 业务日志缺少统一结构，排查问题时难以把“用户操作 -> 状态变化 -> 网络请求 -> 页面反馈”串成完整链路。
- 目前很多日志只能说明“失败了”，无法快速定位“在哪个模块、哪个方法、哪个步骤失败”。

本次设计目标：

- 在保留现有 `AppLogger` 的基础上，建立统一、细粒度、可扩展的日志体系。
- 优先覆盖核心链路，并将页面交互层纳入第一批方案。
- 让日志具备足够的上下文信息，尽量做到“通过日志基本定位问题出现在什么地方”。
- 继续以本地排障为主，日志输出目标保持为控制台 + 本地文件。
- 发布版仍保留尽量详细的日志，但继续执行脱敏、长度裁剪和必要的频控。

非目标：

- 本次不直接接入远端日志平台。
- 本次不追求一次性完成全项目所有页面的百分百覆盖，而是先建立统一规范与可复用接入方式。

## 2. 设计原则

### 2.1 统一结构优先

日志不能继续以“自由拼接字符串”为主。需要统一事件名、统一上下文字段、统一阶段语义，确保跨模块日志可以组合分析。

### 2.2 语义化优先

日志不仅记录“发生了什么”，还要尽量说明：

- 是谁触发的。
- 当前在什么页面、什么流程中。
- 代码执行到了哪个方法、哪个步骤。
- 为什么继续执行、跳过执行或中断执行。

### 2.3 链路可串联

同一次用户操作要通过 `traceId` 串联，尽量覆盖：

- 用户交互。
- 本地校验。
- 状态变更。
- 网络请求。
- 页面反馈。

### 2.4 发布版可保留

发布版也保留详细日志，但需要继续满足以下约束：

- 敏感信息必须脱敏。
- 超长文本必须裁剪。
- 高频日志需要节流、合并或降级，避免日志文件膨胀失控。

## 3. 分层模型

日志分为 5 层，便于快速定位问题归属：

### 3.1 APP 层

记录应用启动与全局运行时事件：

- 依赖初始化开始与完成。
- 全局异常注册。
- 应用启动完成。
- 应用进入前后台。
- 平台层未捕获异常。

### 3.2 ROUTE 层

记录导航与路由决策：

- 页面进入、退出、首帧完成。
- 路由跳转来源、目标。
- 重定向原因。
- 页面恢复与重建。

### 3.3 ACTION 层

记录用户操作与页面交互：

- 按钮点击。
- 提交、删除、确认、取消。
- 支付、上传、分享、拨号。
- 列表项点击、筛选切换、分页加载、下拉刷新。
- 关键输入提交、失焦、校验触发。

### 3.4 STATE 层

记录状态变化与流程分支：

- Controller / Provider 创建与销毁。
- 加载开始、成功、失败。
- 状态前值、后值与变化原因。
- 缓存命中、重复请求、条件不满足导致的跳过。

### 3.5 HTTP 层

记录网络调用全链路：

- 请求发起。
- 请求成功。
- 请求失败。
- 错误分类。
- 关联业务动作与所属页面。

## 4. 统一日志结构

建议所有关键日志结构统一为以下字段集合：

### 4.1 基础字段

- `time`：日志时间。
- `level`：日志级别。
- `layer`：日志层级，如 `APP`、`ROUTE`、`ACTION`、`STATE`、`HTTP`。
- `event`：固定事件名。
- `message`：人类可读说明。

### 4.2 关联字段

- `traceId`：一次用户动作或一次流程链路的标识。
- `sessionId`：当前应用会话标识。
- `requestId`：单次网络请求标识。
- `userId`：当前用户标识。
- `route`：当前页面路由。

### 4.3 代码定位字段

- `module`：业务模块，如 `jobs`、`auth`、`message`。
- `feature`：模块内功能域，如 `post_job`、`login`。
- `controller`：控制器或状态管理对象名。
- `method`：当前方法名。
- `step`：当前执行步骤，如 `enter`、`validate`、`request`、`state_change`、`exit`、`error`。

### 4.4 结果字段

- `result`：`success`、`fail`、`skip`、`cancel` 等。
- `reason`：分支选择、跳过原因或失败原因。
- `durationMs`：耗时。

### 4.5 上下文字段

- `context`：结构化扩展字段。
- `error`：异常文本。
- `stackTrace`：堆栈信息。

## 5. 事件命名规范

日志事件名不再依赖自由文本，统一采用大写蛇形命名，建议规则为：

`<FEATURE>_<ACTION>_<STEP>`

示例：

- `APP_BOOTSTRAP_START`
- `APP_BOOTSTRAP_SUCCESS`
- `ROUTE_REDIRECT_APPLIED`
- `AUTH_RESTORE_START`
- `AUTH_RESTORE_SUCCESS`
- `AUTH_RESTORE_FAIL`
- `POST_JOB_PAGE_ENTER`
- `POST_JOB_SUBMIT_TAP`
- `POST_JOB_VALIDATE_FAIL`
- `POST_JOB_SUBMIT_REQUEST_START`
- `POST_JOB_SUBMIT_REQUEST_FAIL`
- `POST_JOB_SUBMIT_FEEDBACK_SHOWN`

命名要求：

- 同一流程使用同一前缀，便于检索。
- 事件名负责表达“机器可检索语义”。
- `message` 负责表达“人类可读描述”。

## 6. 流程阶段规范

为了让日志能直观看出代码执行到了哪里，关键异步流程统一按阶段记录：

- `enter`：方法进入。
- `input`：输入参数摘要。
- `decision`：分支判断。
- `validate`：本地校验。
- `request`：准备发起请求。
- `response`：收到响应。
- `state_change`：状态更新。
- `feedback`：页面反馈，如 toast、弹窗、跳转。
- `exit`：方法正常结束。
- `error`：方法异常退出。
- `skip`：流程被跳过。

要求：

- 关键方法至少具备 `enter`、`decision/validate`、`request/response`、`state_change`、`exit/error`。
- 被跳过的逻辑必须说明原因，如“已有请求进行中”“缓存命中”“当前角色为空”。

## 7. 作用域与链路串联

### 7.1 Session 级

应用每次启动生成一个 `sessionId`，后续所有日志默认带上该值。

### 7.2 Trace 级

以下场景生成新的 `traceId`：

- 用户点击某个关键操作入口。
- 页面进入后触发自动加载流程。
- 登录、支付、上传、提交表单等长链路业务动作。

同一 `traceId` 需要贯穿：

- 页面交互日志。
- Controller / Provider 日志。
- 网络请求日志。
- 页面反馈日志。

### 7.3 Request 级

已有 `requestId` 继续保留，并补充与 `traceId` 的关联，保证能够从业务动作追到具体请求。

## 8. 页面交互日志策略

本次设计将页面交互纳入第一批覆盖范围，目标接近全量可追踪，但保留必要保护。

### 8.1 必须记录的交互

- 所有会触发业务后果的点击。
- 所有提交、删除、确认、取消动作。
- 上传、支付、分享、拨号。
- 列表项点击且会进入详情或触发业务流程。
- 下拉刷新、分页加载、筛选切换、Tab 切换。

### 8.2 建议记录的输入类交互

- 输入完成。
- 失焦。
- 提交前输入摘要。
- 校验失败结果。

默认不记录每个字符输入，但如果某些场景确实依赖逐字符排查，则需通过节流后记录。

### 8.3 弹层与弹窗

以下事件必须记录：

- 打开。
- 关闭。
- 确认。
- 取消。
- 因校验失败或缺少条件而无法继续。

### 8.4 高频交互保护

以下情况需要频控策略：

- 搜索框频繁输入。
- 拖动、滚动。
- 连续快速切换。

保护方式可选：

- 时间窗口节流。
- 状态变化后再输出。
- 合并多次相同事件为单条摘要。

## 9. 自动采集与手工补点的边界

### 9.1 自动采集负责

- 应用生命周期。
- 页面进入与退出。
- 路由跳转与重定向。
- Provider 更新、异常、销毁。
- HTTP 请求、响应、异常。

### 9.2 手工补点负责

- 用户为什么点了某个动作。
- 当前业务步骤处于什么阶段。
- 为什么跳过或中断流程。
- 页面给用户展示了什么反馈。
- 关键业务参数摘要。

结论：

- 自动采集解决“广覆盖”。
- 手工补点解决“强语义”。
- 两者必须结合，才能满足“通过日志基本定位问题”的目标。

## 10. 现有项目的第一批接入范围

### 10.1 基础设施

- `lib/shared/logging/app_logger.dart`
- `lib/shared/network/interceptors/app_log_interceptor.dart`
- `lib/shared/network/api_client.dart`
- `lib/shared/network/dio_factory.dart`

### 10.2 应用与路由

- `lib/main.dart`
- `lib/app/router/app_router.dart`
- `lib/app/app.dart`

### 10.3 鉴权与会话

- `lib/features/auth/application/auth_session_provider.dart`
- 登录、角色选择、会话恢复相关页面与流程。

### 10.4 业务控制器优先级

第一批优先处理：

- 岗位发布相关控制器与页面。
- 消息会话相关控制器与页面。
- 订单、支付、上传相关控制器与页面。
- 黑名单、收藏、简历编辑等高频交互页面。

## 11. 实现形态

### 11.1 保留底层输出

继续使用现有 `AppLogger` 作为底层实现，负责：

- 控制台输出。
- 文件落盘。
- 脱敏。
- 长度裁剪。
- 文件轮转。

### 11.2 增加统一日志门面

新增语义化日志助手，示例职责如下：

- `AppFlowLog`：应用启动、生命周期。
- `RouteLog`：页面进入、退出、跳转、重定向。
- `ActionLog`：点击、提交、确认、取消、输入提交。
- `StateLog`：状态变化、创建、销毁、跳过。
- `HttpFlowLog`：业务动作与请求关联。

### 11.3 增加作用域上下文

需要一套上下文注入机制，用于自动附加：

- `sessionId`
- `traceId`
- `route`
- `userId`
- `module`
- `feature`

理想效果：

- 业务代码声明一次上下文。
- 后续日志自动继承，不必每次手工拼接完整字段。

### 11.4 增加自动观察能力

- 增加 GoRouter 日志观察器。
- 增加 Riverpod `ProviderObserver`。
- 增加页面生命周期日志助手。

## 12. 发布版与隐私策略

发布版保留尽量详细日志，但必须遵守：

- token、authorization、password、secret 等字段统一脱敏。
- 身份证号、手机号、邮箱等个人信息后续应纳入更细粒度脱敏名单。
- 响应体与异常文本继续做长度裁剪。
- 对明显无排障价值的超高频事件执行频控。

## 13. 分阶段实施计划

### 阶段一：增强日志底座

- 扩展统一日志结构。
- 增加 `sessionId`、`traceId`、`event`、`layer`、`module`、`method`、`step` 等字段支持。
- 增加语义化门面与上下文能力。

### 阶段二：补自动采集

- 路由日志增强。
- 页面生命周期日志。
- Riverpod 观察日志。
- 网络请求与业务动作关联。

### 阶段三：补核心业务语义

- 启动链路。
- 登录态与会话恢复。
- 岗位发布。
- 消息会话。
- 订单、支付、上传。
- 其他高频交互页面。

### 阶段四：逐步扩展到全项目

- 按统一日志助手继续补模块。
- 收敛命名规范。
- 根据真实日志文件大小与排障收益，调整频控与字段粒度。

## 14. 验收标准

完成第一轮实施后，应满足以下条件：

- 能从日志明确看到应用启动、路由跳转、登录态恢复、核心请求、关键交互的前后链路。
- 某个核心流程失败时，能从日志快速定位到模块、方法和步骤。
- 同一次用户操作具备 `traceId` 串联。
- 发布版日志仍可用于排障，且敏感信息不会明文输出。
- 新增日志方式统一，不依赖自由拼接字符串。

## 15. 风险与控制

### 15.1 日志过量

风险：

- 文件体积增长快。
- 有效信息被噪音淹没。

控制：

- 高频交互节流或摘要化。
- 统一字段，减少无意义重复文本。

### 15.2 业务代码侵入过多

风险：

- 接入成本高。
- 后续维护负担大。

控制：

- 通过日志助手和作用域上下文降低样板代码。
- 自动采集和手工补点搭配使用。

### 15.3 隐私泄露

风险：

- 请求头、用户信息、证件信息、手机号等写入日志。

控制：

- 强制脱敏。
- 增加敏感字段名单。
- 必要时对部分字段只保留摘要。
